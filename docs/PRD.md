## 1. 产品概述

**Fuel Frog Panic** 是构建于 EVE Frontier 之上的一个**链上竞技任务平台**。  
它将游戏内最枯燥的后勤动作——"送燃料（Fuel）"——改造成一场**10 分钟限时竞速赛**，通过：

- 实时链上计分
- LUX 奖池激励
- 游戏内浮窗（In-game Overlay）反馈
- 多角色协作分工

让"送油"从无名的义务劳动变成有成就感、有即时收益的竞技体验。

---

## 2. 背景与问题定义

### 2.1 为什么"燃料"是游戏核心？

在 EVE Frontier 中，**燃料（Fuel）是文明的血液**，所有基础设施都依赖其运转：

| 设施 | 断油后果 |
|---|---|
| **星门 (Gates)** | 无法在星系间跳跃，玩家被困死在当前星区 |
| **智能存储单元 (SSU)** | 无法处理任何交易、制造或合约执行 |
| **防御炮塔 (Turrets)** | 持续烧油，断电后家门口毫无防范 |

站点油量由 **`fill_ratio`（填充率）** 表示，三级告警如下：

- 🔴 **红色（< 20%）**：极度危险，随时停机
- 🟡 **黄色（< 50%）**：一般警告
- 🟢 **绿色（> 50%）**：安全运行

### 2.2 "送油"的供应链结构

送油不是一个人能完成的，涉及完整的工业流水线：

| 角色 | 职责 | 痛点 |
|---|---|---|
| **采集 (Collector)** | 寻找燃料气云/矿石 → 开采 → 装舱 | 枯燥等待、易被偷袭 |
| **运输 (Hauler)** | 装载燃料 → 规划航线 → 规避风险区 → 对接站点 | 飞船笨重、跳跃繁复、无战斗爽感 |
| **护航 (Escort)** | 扫描威胁 → 保护运输船 | 大量时间"发呆"，若无敌人则毫无感知 |

### 2.3 核心痛点总结

| 维度 | 现状 |
|---|---|
| **反馈缺失** | 完成后链上只跳出一个小数字，没有成就感 |
| **利益分配模糊** | 后勤玩家是"无名英雄"，难以实时看到劳动换算的收益 |
| **协作脱节** | 队长喊"快去送油"，但队员不知道哪个站最急、能拿多少奖励 |
| **高风险低回报** | 运输最后一公里被海盗击毁，所有辛苦化为乌有且无人知晓 |

## 3. 目标用户

| 用户角色 | 描述 | 核心诉求 |
|---|---|---|
| **站点所有者 / 联盟盟主** | 持有 SSU 或星门的玩家，担心设施停机 | 快速吸引外部玩家来救急，防止防线崩溃 |
| **竞技型玩家 / 战队队长** | 追求竞技乐趣与 LUX 收益 | 找到合适的对手/站点，用技术和协作赢走奖池 |
| **新手 / 普通联盟成员** | 想参与却不知道干什么、干完没有反馈 | 清晰的任务目标、即时的成就感、可见的收益 |

## 4. 核心功能模块

### 4.0 登录：连接 EVE Frontier 内置钱包

> 平台的一切功能（查看比赛、组队、付费、链上计分）都以**已连接钱包的用户身份**为前提。用户首次进入站点时，必须完成钱包连接才能继续操作。

---

#### 4.0.1 登录流程

```
用户访问 Fuel Frog Panic
      ↓
检测是否已有活跃的钱包连接会话
      ↓
      ├─ 已连接 → 直接进入星系热力图（4.1）
      │
      └─ 未连接 → 展示「连接钱包」引导页
                        ↓
              点击 [Connect EVE Frontier Wallet]
                        ↓
              唤起 EVE Frontier 内置钱包授权弹窗
                        ↓
              用户在钱包内确认授权
                        ↓
              连接成功 → 读取钱包地址与 LUX 余额
                        ↓
              进入星系热力图（4.1）
```

---

#### 4.0.2 连接状态展示

连接成功后，页面右上角**持久显示**钱包状态栏：

| 字段 | 说明 |
|---|---|
| **钱包地址** | 显示前 6 位 + `...` + 后 4 位（如 `0x1a2b...cd3e`） |
| **LUX 余额** | 实时显示当前可用余额，余额不足时标红提示 |
| **断开连接** | 点击可退出当前钱包会话 |

---

#### 4.0.3 未连接状态的功能限制

| 功能 | 未连接 | 已连接 |
|---|---|---|
| 查看星系热力图与站点油量 | ✅ 可浏览 | ✅ |
| 查看加油比赛大厅 | ✅ 可浏览 | ✅ |
| 发起组队 / 加入战队 | ❌ 需连接钱包 | ✅ |
| 支付入场费 / 参加比赛 | ❌ 需连接钱包 | ✅ |
| 链上计分归因 | ❌ 无法识别身份 | ✅ |
| 领取奖励 | ❌ 需连接钱包 | ✅ |

> 浏览型功能对未登录用户开放，降低进入门槛；涉及资产或身份的操作一律要求钱包连接。

#### 4.1.1 数据来源

- **`NetworkNode`** 是 Sui 链上的共享对象（Shared Object），任何 dApp 均可公开读取
- 通过合约的 `fuel_quantity` + `fuel_max_capacity` 实时计算 `fill_ratio`
- 通过 `reveal_location` 函数获取站点坐标
- 后端持续监听 `NetworkNodeCreatedEvent` 扫描全星系新增站点

---

#### 4.1.2 核心视觉：地图双图层设计

地图不是静态的点集，而是**实时"动起来"的热力场**，由两个叠加图层构成：

**图层一：油量底图（Base Layer）**

站点按 `fill_ratio` 着色，直接传达紧急度：

| 颜色 | 条件 | 含义 |
|---|---|---|
| 🔴 红色 | `fill_ratio` < 20% | 极度危险，极高权重计分 |
| 🟡 黄色 | 20% ≤ `fill_ratio` < 50% | 一般告警 |
| 🟢 绿色 | `fill_ratio` ≥ 50% | 安全运行 |

**图层二：比赛脉冲（Match Pulse Layer）**

凡是已有比赛正在报名或进行中的站点，在其坐标点外圈叠加**不断向外扩散的金色光圈**：

- **光圈半径**：与当前奖池（Prize Pool）金额正比——奖池越高，光圈越大
- **闪烁频率**：与比赛开始/结束的剩余时间负相关——越临近截止，闪烁越快
- 两层叠加效果：玩家一眼就能识别出"红色 + 剧烈闪烁 + 大光圈"的高价值目标站

**左侧：交互式地图（Interactive NodeMap）**

- 点击**无比赛的红色站点** → 弹出 `[发起比赛]` 操作入口
- 点击**已有比赛的站点** → 弹出 `[立即加入]` 按钮，并显示当前有多少队在抢
- 支持侧边栏筛选（按奖金最高/紧急度最高），地图自动缩放定位至目标站点

**右侧：加油比赛大厅（Fueling Match Board）**

- 比赛卡片按"奖池金额 × 紧急度"加权排序，最值得参与的置顶
- 每张比赛卡片展示以下字段：

| 字段 | 示例 |
|---|---|
| 目标站（颜色标识） | 🔴 SSU-Frontier-7 |
| **总奖池** | **1,200 LUX** |
| **入场费** | **100 LUX / 人** |
| **战队规模（必选展示）** | 最少 **1** 支、最多 **N** 支（**N 为本局可配置「战队上限」**，如 **10**）；当前已报名 **3 / N** 支（含「已付费锁定」与「仅组队未付」可区分时用标签区分） |
| **开赛门槛（必选展示）** | 与规则一致的文字说明，例如「满 **N** 支战队且均已付费即开」（N 即本局战队上限）或「未达满员时：至少 **1** 支战队满 **3** 人则 **3 分钟**后强制开赛」 |
| 参与战队 | A 队 vs B 队（2/2 满员）|
| 倒计时 | 02:34 |

> **UI 强制要求（比赛队伍参数）**  
> 每一局加油比赛在创建时由系统写入的**队伍相关规则参数**（包括但不限于：**最少参赛战队数**、**最多参赛战队数/战队上限**、以及与本局绑定的**开赛触发模式**要点），必须在**比赛大厅卡片**、**站点详情弹窗**、以及进入**组队大厅**前的确认区中**一致展示**，不得仅存在于后端或文案省略。玩家在未进入钱包支付前，应能仅凭 UI 读懂「这局需要几支队、最多几支队、当前进度到哪一步」。

> **战队上限（不固定为 4）**  
> **最多参赛战队数不是固定 4 支**：可按单局配置为 **4、10、16 或更高**，以满足不同规模联盟战或练兵房。**建议**仍由平台设定**可配置的上限**（或每局单独 cap），用于控制单局匹配时长、服务端/合约侧列表规模与 UI 可读性——这是**工程与运营策略**，不是「业务上只能 4 支队」。v1 自动生成紧急赛时的默认值（若有）仅作默认，**玩家可见文案始终以本局 `N` 为准**。

#### 4.1.5 比赛发布：发布者规则

> **v1 版本**：比赛仅由**平台官方**自动发布，玩家无法手动创建比赛。以下为完整的三类发布者设计，供后续版本迭代参考。

**A. 系统/平台自动生成（v1 唯一发布方）**

- 后端持续通过 Sui RPC 轮询全星系 `fill_ratio`
- 触发条件：公共/中立站点油量低于 **20%（红色警戒）**
- 自动在加油比赛大厅生成"紧急救援比赛"，并预设开赛规则（战队上限、最低人数等）
- 奖池由玩家入场费 + 平台官方补贴构成

**B. 站点所有者（后续版本）**

- 在地图选中自己的站点 → 点击「发起救援」
- 可向奖池额外注入（Inject）一笔 LUX 作为悬赏金
- 动机：防止设施 Offline 导致防御失效或生产中断

**C. 普通玩家/战队队长（后续版本）**

- 任何玩家可在地图上选中 3–5 个站点，创建"补给竞技房"
- 用途：联盟练兵 / 技术对赌
- 奖池完全由战队入场费构成

**悬赏加码功能（后续版本）**

- 站点所有者可追加 LUX 到奖池，地图上该站的金色光圈半径瞬间扩大，引发全星系玩家注意

#### 4.1.6 用户操作漏斗：从"看"到"干"

```
① 扫视 (Scanning)
   打开地图 → 寻找闪烁最剧烈、颜色最红的点（钱多且急）
        ↓
② 筛选 (Filtering)
   点击侧边栏「奖金最高」→ 地图自动缩放至目标站点
        ↓
③ 决策 (Decision)

        ↓
④ 行动 (Commitment)
   点击 [Join & Pay] → 钱包签名 → 正式进入 10 分钟竞技
```

### 4.2 组队大厅（Squad Lobby）

> 玩家在加油比赛大厅选定比赛后，进入组队大厅。这里只做一件事：**组好队**。玩家可以发起一支新战队，也可以直接加入已存在的战队。

> **UI 要求**：组队大厅页顶或比赛标题区须**持续展示**与本局一致的 **最少/最多战队数**、**当前已报名战队数**（及与 4.1.3 相同的开赛门槛摘要），但展示形态应保持**轻量化**，优先使用一条状态条或标题区副信息，避免形成独立的大面积信息卡片。

---

#### 4.2.1 发起组队

房主在此创建一支新战队，设定以下参数后等待队员加入：

| 参数 | 可选范围 | 说明 |
|---|---|---|
| **队伍人数上限** | 3–8 人 | 3 人为可运转最小单位（采集 + 运输 + 护航各一）；8 人上限提供更高容错率 |
| **角色槽位分配** | 自由分配 | 房主可设置多个"运输"槽，策略性强化某一环节 |

设定完成后，房主点击 **`[创建战队]`**，战队卡片出现在组队大厅列表中供他人加入。

---

#### 4.2.2 加入战队

大厅列表展示当前所有招募中的战队卡片，每张卡片显示：

| 字段 | 说明 |
|---|---|
| **当前人数 / 上限** | 如 `3 / 8` |
| **角色缺口** | 当前缺少哪些角色（如"缺 2 名运输"），帮助玩家快速决定要不要加 |
| **队长荣誉值** | 队长的信誉星级（1–5 星） |

点击任意卡片 → 选择角色 → 点击 **`[加入]`** 完成入队。

---

#### 4.2.3 角色说明

选择角色时弹出角色说明卡：

| 角色 | 核心职责 | 对队伍的价值 |
|---|---|---|
| **采集 (Collector)** | 寻找燃料气云并开采，是流水线源头 | 决定燃料总供给上限 |
| **运输 (Hauler)** | 把燃料从采集点运到目标 `NetworkNode` | 直接触发 `deposit_fuel()`，贡献得分 |
| **护航 (Escort)** | 扫描威胁，保护运输船免遭拦截 | 防止运输船被击毁导致运输白费 |

---

#### 4.2.4 队长锁定

所有角色槽位填满后，队长点击 **`[Lock]`** 锁定队伍阵容：

- 槽位全部冻结，不可再更改角色或成员
- 战队状态变为"已就绪（Ready）"，可以开始报名参加加油比赛

> 此时**不涉及任何付费**。锁定只是确认了队伍的人员构成。

---

#### 4.2.5 参加加油比赛与付费

锁定后的战队可以前往**加油比赛大厅**报名参赛。报名时需满足比赛的准入要求，例如：

- 队伍人数达到比赛规定的最低人数
- 角色配置符合比赛要求（如必须包含至少 1 名运输手）

满足条件后，队长点击 **`[Join & Pay]`**，触发付费确认弹窗：

- 队长一次性按人头总付入场费
- 付款成功后，全队地址自动注册进该场比赛的白名单
- 白名单注册是后续**链上归因计分三重过滤**的第一道验证

> 不符合准入要求的战队，`[Join & Pay]` 按钮置灰，并提示具体未达标项。

---

#### 4.2.5 开赛触发（系统自动）

开赛时机由**发起比赛时预设的规则**决定，与队长无关。目前支持两种模式：

| 模式 | 触发条件 | 等待时长 | 说明 |
|---|---|---|---|
| **满员即刻触发（Dynamic Trigger）** | 房间内所有战队均已付费锁定 | 自动开启 **30 秒**准备倒计时 | 最常用模式；给玩家最后时间在 EVE 里调整飞船位置 |
| **最低人数强制启动（Minimum Threshold）** | 房间未满，但已达到最低参赛人数（如 1 队 3 人） | 自动开启 **3 分钟**倒计时 | 防止因招募缓慢导致比赛长期无法开始 |

两种模式下，倒计时结束后**强制开赛**——无论玩家是否在操作，全服计分正式开启，切换至比赛浮窗（MatchRunning）。

---

### 4.3 比赛进行（MatchRunning）

> 这是整个产品的**多巴胺核心**。玩家一边在 EVE 客户端里操作飞船，一边盯着浮窗里的实时战报。10 分钟内，反馈密度要比任何一款传统后勤游戏都高出一个数量级。

---

#### 4.3.1 开赛触发逻辑

| 模式 | 触发条件 | 等待行为 |
|---|---|---|
| **满员即刻开赛（Dynamic Trigger）** | 所有战队全员付费并锁定角色 | 自动开启 **30 秒**最终准备倒计时，无法取消 |
| **最低人数强制启动（Minimum Threshold）** | 至少 1 队达到 3 人，但房间长时间未满员 | 系统开启 **3 分钟**公开倒计时，期间仍可加人 |
| **单队挑战赛模式** | 报名截止时只有一支队伍 | 对手变为"站点状态"本身；10 分钟内将 `fill_ratio` 提升至 ≥ 80% 判定"大成功"，额外获得 10% 补贴奖励 |

#### 4.3.3 Optimistic UI 机制

链上事务确认存在 1–3 秒延迟，为避免玩家感知"操作无响应"，采用**乐观预更新**策略：

```
玩家在 EVE 中点击 deposit_fuel()
        ↓
浮窗进度条立刻以"虚线跳动"动画显示预期增量（灰色虚线段）
        ↓
后端收到 FuelEvent(DEPOSITED) 确认
        ↓
虚线变实线，得分数字正式跳动 + 弹幕滚出
```

- 若交易失败（超时/被拦截），虚线段回退并显示 ⚠️ "本次运输被中断"
- 该机制将玩家的主观等待感从 3 秒压缩至 0 秒

---

#### 4.3.4 站点进度条交互细节

每个目标站点的进度条有以下视觉状态：

| 状态 | 表现 |
|---|---|
| `fill_ratio` < 20%（红） | 进度条血红色 + 闪烁动画，右侧显示 "⚠️ 濒危 — 3.0x 加成！" |
| `fill_ratio` 20%–50%（黄） | 进度条橙黄色，右侧显示 "1.5x 加成" |
| `fill_ratio` > 50%（绿） | 进度条绿色，右侧显示 "1.0x" |
| 刚收到燃料（任意） | 进度条末端出现 0.5 秒的"注入涟漪"动画，强调实时变化 |

---

#### 4.3.5 最后 90 秒：Panic 模式

进入 Panic 阶段时，UI 发生全面变化，制造极限紧迫感：

**视觉变化**

- 浮窗整体边框开始**橙红色高频脉冲闪烁**（频率随倒计时加快）
- 倒计时字体放大 1.5 倍，颜色切换为血红
- 所有站点进度条（含绿色站）全部进入闪烁状态，提示"最后的补给机会"
- 浮窗顶部出现横幅："🔥 PANIC MODE — 全部得分 ×1.5！"

**音效变化**

- 背景出现低频心跳声，每秒一次
- 每次 `deposit_fuel()` 成功触发，音效从普通提示音升级为更响亮的"电流充能"音效

**得分机制变化**

- 所有送油得分额外乘以 **1.5x**（与紧急度权重叠加计算）
- 落后队在此阶段完成一次大宗运输，可能直接逆转排名

---

#### 4.3.6 比赛状态机（MatchStatus 状态迁移）

```
Lobby（等待报名）
    ↓ 满员 or 最低人数倒计时归零
PreStart（30 秒准备）
    ↓ 倒计时归零
Running（10 分钟竞技）
    ↓ 计时结束
    ├── 前 8.5 分钟：Normal Mode（标准计分）
    └── 最后 90 秒：Panic Mode（1.5x 加成）
Settling（结算中，约 10–30 秒）
    ↓ 链上结算完成
Settled（已结算，可领奖）
```

后端通过定时任务每秒检查所有 `MatchStatus = Running` 的房间，到达时间节点后调用合约的 `end_match(match_id)` 触发链上结算。

### 4.4 结算与奖励分配（Settlement & Rewards）

> 结算页是玩家整场体验的**情绪终点**。设计目标：让赢家感到扬眉吐气，让输家感到"下次还有机会"，让所有人都清楚地看到自己的链上贡献换来了多少真实收益。

---

#### 4.4.1 结算触发流程

```
比赛计时器归零
      ↓
后端调用 end_match(match_id)，链上冻结本局得分快照
      ↓
Settling 状态（约 10–30 秒）：
  · 浮窗显示"⏳ 正在结算链上数据..."
  · 进度条缓慢加载，给玩家一个"仪式感"的等待
      ↓
Settled 状态：
  · 浮窗切换为"战报页"
  · 奖励金额自动发放至胜者钱包
```

---

#### 4.4.2 奖池构成

一场比赛的总奖池由以下部分组成：

| 来源 | 说明 | v1 是否启用 |
|---|---|---|
| **战队入场费** | 各队队长按人头总付，全额进入奖池（主要来源） | ✅ |
| **平台官方补贴** | 系统自动生成比赛时由平台注入的启动资金 | ✅ |
| **站点所有者悬赏金** | 站主为吸引救援而额外注入的 LUX | 后续版本 |

**示例**：A 队 5 人 + B 队 5 人，入场费 100 LUX/人，平台补贴 200 LUX
$$
\text{总奖池} = (5 + 5) \times 100 + 200 = 1{,}200 \text{ LUX}
$$

---

#### 4.4.3 队伍级分配（第一层：队与队之间）

无论有多少支战队参赛，**最多只评前 3 名**，名次之外的队伍不参与奖池分配。分配比例根据实际参赛队伍数量分四种情况：

---

**情况一：仅 1 支队伍（单队挑战赛模式）**

不存在队间竞争，改为挑战固定目标：

| 结果 | 奖励 |
|---|---|
| 10 分钟内将目标站点 `fill_ratio` 提升至 **80% 以上** | 该队获得奖池 **100%**，额外获得平台 10% 补贴加成 |
| 未达成目标 | 入场费**不退还**，平台补贴回收 |

---

**情况二：2 支队伍**

| 排名 | 比例 | 示例（总奖池 1,200 LUX）|
|---|---|---|
| 🥇 冠军队 | **70%** | 840 LUX |
| 🥈 亚军队 | **30%** | 360 LUX |

---

**情况三：3 支队伍**

| 排名 | 比例 | 示例（总奖池 1,200 LUX）|
|---|---|---|
| 🥇 冠军队 | **60%** | 720 LUX |
| 🥈 亚军队 | **30%** | 360 LUX |
| 🥉 季军队 | **10%** | 120 LUX |

> 当前默认值，后续可调整。

---

**情况四：3 支以上队伍**

仍只取**前 3 名**，按情况三的比例（60/30/10）分配。第 4 名及以后的队伍**不获得任何奖励**，其入场费已计入总奖池。

| 排名 | 比例 |
|---|---|
| 🥇 冠军队 | **60%** |
| 🥈 亚军队 | **30%** |
| 🥉 季军队 | **10%** |
| 第 4 名及以后 | **0%** |

---

**各情况汇总**

| 参赛队数 | 🥇 冠军 | 🥈 亚军 | 🥉 季军 | 备注 |
|---|---|---|---|---|
| 1 队 | 100%（达标） | — | — | 未达标则奖池不发放 |
| 2 队 | 70% | 30% | — | — |
| 3 队 | 60% | 30% | 10% | 默认值，可调 |
| ≥ 4 队 | 60% | 30% | 10% | 第 4 名起不参与分配 |

#### 4.4.4 个人级分配（第二层：队内成员之间）

队伍获得的奖励金额，再按**每位成员在链上实际送油的得分贡献比例**进行二次分配：

$$
\text{个人奖励} = \text{队伍奖励} \times \frac{\text{个人得分}}{\text{队伍总得分}}
$$

**计算示例**（🅐 队获得 840 LUX，队内 3 名成员）：

| 成员 | 角色 | 个人得分 | 贡献比例 | 个人奖励 |
|---|---|---|---|---|
| 玩家 B | 运输 | 2,100 pts | 48.8% | **410 LUX** |
| 玩家 A | 采集 | 1,800 pts | 41.9% | **352 LUX** |
| 玩家 C | 护航 | 400 pts | 9.3% | **78 LUX** |
| **合计** | — | **4,300 pts** | 100% | **840 LUX** |

**关键机制**：

- 得分数据完全来自链上 `FuelEvent`，无法造假
- 不送油 = 链上得分为 0 = 个人奖励为 0，彻底杜绝挂机蹭分
- 护航（Escort）角色无法直接触发 `deposit_fuel()`，其得分来源需在计分系统中单独定义（见第 5 章）

## 5. 计分系统

> 计分系统是整个竞技公平性与策略深度的基石。核心设计原则：**"送得多"只是基础，"送得对、送得急、送得准时"才是胜负手。**

---

### 5.1 基础计分公式

每一次 `deposit_fuel()` 触发 `FuelEvent(DEPOSITED)` 后，后端立即计算本次得分：

$$
\text{delivery\_score} = (\text{new\_quantity} - \text{old\_quantity}) \times \text{urgency\_weight(node)} \times \text{panic\_multiplier}
$$

| 变量 | 说明 | 取值 |
|---|---|---|
| `new_quantity - old_quantity` | 本次实际注入的燃料数量（链上差值） | 由合约 `FuelEvent` 提供 |
| `urgency_weight(node)` | 该站点在**注入时刻**的紧急度权重 | 见 5.2 |
| `panic_multiplier` | 是否处于 Panic 模式 | 正常：1.0；Panic 阶段：1.5 |

**关键设计细节**：权重以**注入时刻的 `fill_ratio`** 为准，而非注入后的值。这防止了"故意少量多次注入以维持红色状态从而刷高分"的作弊行为。

---

### 5.2 紧急度权重（Urgency Weight）

权重反映"这口油对宇宙来说有多值钱"，与 `fill_ratio` 阈值挂钩：

| 站点状态 | `fill_ratio` 区间 | 权重系数 | 设计意图 |
|---|---|---|---|
| 🔴 **极度危险** | < 20% | **3.0x** | 救活濒死站点，价值最高 |
| 🟡 **一般告警** | 20% – 50% | **1.5x** | 预防性补给，中等价值 |
| 🟢 **安全运行** | ≥ 50% | **1.0x** | 锦上添花，基础价值 |

**策略含义**：

- 一个队若能专注"救红站"，哪怕送油总量只有对手的一半，依然可能凭借 3.0x 权重反超
- 对手若一直给绿站"刷量"，实际得分效率极低——这逼出了真正有价值的协作决策

---

### 5.3 冲刺加成（Panic Multiplier）

| 阶段 | 时间段 | 得分乘数 |
|---|---|---|
| **Normal Mode** | 比赛开始 → 剩余 90 秒 | 1.0x |
| **Panic Mode** | 最后 90 秒 | **1.5x** |

Panic 加成与紧急度权重**叠加计算**。极端情况下：

$$
\text{最高单次得分系数} = 3.0 \times 1.5 = \textbf{4.5x}
$$

> 示例：在最后 90 秒给一个 `fill_ratio = 8%` 的红色站点送入 500 单位燃料：  
> 得分 = 500 × 3.0 × 1.5 = **2,250 pts**，足以一次性逆转大多数局面。

---

### 5.4 得分计算示例

假设某局比赛中，🅐 队和 🅑 队的行动如下：

| 行动 | 队伍 | 送油量 | 站点状态 | 时间段 | 得分 |
|---|---|---|---|---|---|
| 玩家B 送油 SSU-7 | 🅐 | 200 | 🔴 8% | Normal | 200 × 3.0 × 1.0 = **600** |
| 玩家X 送油 Gate-12 | 🅑 | 500 | 🟢 72% | Normal | 500 × 1.0 × 1.0 = **500** |
| 玩家B 送油 SSU-3 | 🅐 | 150 | 🔴 15% | Panic | 150 × 3.0 × 1.5 = **675** |
| 玩家Y 送油 Gate-12 | 🅑 | 400 | 🟢 80% | Panic | 400 × 1.0 × 1.5 = **600** |

结果：🅐 队以 1,275 pts 对 1,100 pts 胜出——尽管 🅑 队总送油量（900）远超 🅐 队（350）。

---

### 5.5 个人贡献得分与队伍总分的关系

- **队伍总分**：队内所有成员 `delivery_score` 的累加
- **个人贡献比**：`个人得分 / 队伍总得分`，用于 Tier 2 奖励分配（见 4.4.3）
- **MVP 评定**：全场（跨队伍）个人得分最高者

个人贡献数据 100% 来自链上 `FuelEvent`，不可篡改，确保分配公平。

## 6. 付费模式

> 付费设计的核心目标：**最大化奖池规模**（让参与更值得）+ **最小化队员摩擦**（降低入场门槛）+ **最大化透明度**（让每个人都清楚"我投了多少、能拿多少"）。

---

### 6.1 入场费结构

#### 收费单位

- 以**人头**计费，单价固定为 **100 LUX / 人**（可由比赛发布者配置）
- **队长**是唯一的链上付款人，一次性按当前队伍人数总付

| 队伍人数 | 队长支付总额 | 奖池贡献 |
|---|---|---|
| 3 人 | 300 LUX | 全额进入奖池 |
| 5 人 | 500 LUX | 全额进入奖池 |
| 8 人 | 800 LUX | 全额进入奖池 |

#### 奖池实时累加

当双队都已付费时，奖池金额 = 双方入场费总和 + 额外悬赏金。玩家在 Lobby 中可看到奖池数字实时更新，产生"快进来，钱越来越多"的视觉刺激。

---

### 6.2 队员零成本入场

队员的入场体验设计为**零摩擦**：

```
队员操作路径：
① 在战队列表中点击「加入队伍」
```

## 7. 链上归因逻辑（On-chain Attribution）

> 归因逻辑是整个平台可信度的基石。**所有得分数据必须 100% 来自链上，不依赖任何玩家自报或服务端信任假设。**

---

### 7.1 唯一数据源

- **合约模块**：`world::fuel`
- **事件**：`FuelEvent(DEPOSITED)`
- **触发条件**：玩家在 EVE 中执行 `deposit_fuel()` 指令并成功上链后，合约自动发射此事件

`FuelEvent` 携带的关键字段：

| 字段 | 说明 |
|---|---|
| `assembly_id` | 被加油的 NetworkNode 对象 ID（唯一标识目标站点） |
| `old_quantity` | 加油前的燃料数量 |
| `new_quantity` | 加油后的燃料数量 |
| `tx_digest` | 发射该事件的交易哈希 |

---

### 7.2 归因流程（完整时序）

```
① 链上事件捕获
   后端订阅 Sui 事件流，实时接收 FuelEvent(DEPOSITED)
        ↓
② 交易溯源
   通过 sui_getTransactionBlock(tx_digest) 获取：
   · sender（发起交易的钱包地址）
   · timestamp（交易时间戳）
        ↓
③ 三重过滤（全部通过才计分）
   ✅ Filter 1 — 白名单校验：sender ∈ 本局已注册玩家集合
   ✅ Filter 2 — 站点校验：assembly_id ∈ 本局目标站点列表
   ✅ Filter 3 — 时间窗口：timestamp ∈ [match_start, match_end]
        ↓
④ 得分计算
   score = (new_quantity - old_quantity) × urgency_weight × panic_multiplier
        ↓
⑤ 实时推送
   · 更新玩家得分 → 通过 WebSocket 推送至浮窗
   · 触发弹幕消息 → 发送至所有同局玩家浮窗
   · 更新站点 fill_ratio 缓存 → 下一次权重计算使用
```

---

### 7.3 三重过滤详解

#### Filter 1：白名单校验

- 白名单在队长支付入场费时，由后端一次性将全队地址写入本局记录
- 作用：**防止非参赛者的链上动作被计入本局得分**（例如旁观者碰巧也给同一站点加了油）
- 数据结构：`match_id → Set<wallet_address>`

#### Filter 2：站点校验

- 本局目标站点列表在比赛创建时固定，存储为 `match_id → Set<assembly_id>`
- 作用：**防止玩家给非目标站点加油来刷分**（比如给自己的私人站点加油）
- 注意：一局比赛可设 1–5 个目标站点，每个站点按各自的 `fill_ratio` 独立计算权重

#### Filter 3：时间窗口

- `match_start`：合约 `start_match()` 被调用的链上时间戳
- `match_end`：`end_match()` 被调用的链上时间戳
- 作用：**防止比赛开始前的"预热"加油或结束后的延迟交易被计入**
- 边界处理：时间戳对比以链上区块时间为准，非服务器时间

---

### 7.4 防作弊设计

| 潜在作弊行为 | 应对机制 |
|---|---|
| 未参赛玩家代为加油 | Filter 1 白名单拦截 |
| 给非目标站加油刷分 | Filter 2 站点校验拦截 |
| 比赛结束后继续提交 | Filter 3 时间窗口拦截 |
| 权重操控（少量多次维持红色） | 权重以注入**时刻**的 `fill_ratio` 计算，不允许逆向推算 |
| 得分数据造假 | 全部数据来自链上事件，服务端只读不写 |

### 8.3 后端服务

#### 全局节点扫描服务（Node Scanner）

- 持续监听 `NetworkNodeCreatedEvent`，维护全星系站点列表
- 每 **5 秒**批量查询所有已知 `NetworkNode` 的 `fuel_quantity` + `fuel_max_capacity`
- 计算并缓存 `fill_ratio`，触发自动任务生成逻辑：

```
if fill_ratio < 0.20 AND node.is_public AND no_active_match:
    auto_create_match(node_id, bounty=platform_subsidy)
    broadcast_to_all_clients("紧急任务！[站名] 油量告急！")
```

#### 事件监听服务（Event Listener）

- 订阅 Sui 链上 `world::fuel` 模块的事件流
- 捕获 `FuelEvent(DEPOSITED)` 后立即执行归因三重过滤（见第 7 章）
- 通过内部消息队列推送至 API Server，触发 WebSocket 广播

#### 比赛状态机服务（Match State Manager）

- 每秒检查所有 `MatchStatus = Running` 的比赛
- 负责以下状态迁移：

| 迁移 | 触发条件 | 链上动作 |
|---|---|---|
| Lobby → PreStart | 满员 or 最低人数倒计时归零 | 无（纯后端状态） |
| PreStart → Running | 30 秒倒计时归零 | 调用 `start_match(match_id)` |
| Running → Panic | 剩余时间 ≤ 90 秒 | 无（推送前端切换视觉模式）|
| Running → Settling | 10 分钟计时结束 | 调用 `end_match(match_id)` |
| Settling → Settled | 链上结算 tx 确认 | 自动发放奖励至获胜队伍钱包 |

## 11. 链上 / 链下边界（On-chain vs Off-chain）

> 本章为研发统一口径，明确哪些数据**必须上链**（不可篡改、需公开验证）、哪些**存后端数据库**即可（高频读写、无需链上共识），避免过度上链拖慢性能或欠上链导致结果可信度不足。

---

### 11.1 必须上链的数据

这些数据直接涉及**资产归属、结果公正性或可验证性**，必须以链上交易的形式存在：

| 数据 | 链上形式 | 原因 |
|---|---|---|
| **玩家送油动作** | `deposit_fuel()` 交易 + `FuelEvent(DEPOSITED)` | 唯一可信的计分数据源，不可由后端伪造 |
| **站点燃料状态** | `NetworkNode` 对象的 `fuel_quantity` / `fuel_max_capacity` | EVE Frontier 原生链上数据，平台直接读取 |
| **入场费支付** | 队长钱包发起的 LUX 转账交易 | 涉及真实资产，必须链上留证 |
| **参赛白名单注册** | 比赛 Object 内的地址列表写入 | 白名单是归因计分的信任根，必须防篡改 |
| **比赛开始 / 结束** | `start_match()` / `end_match()` 交易 | 锁定计分时间窗口，链上可审计 |
| **奖励发放** | 结算合约向获胜队钱包的 LUX 转账 | 涉及真实资产出账，必须链上留证 |
| **MVP / 勋章 NFT** | 链上 NFT Mint | 荣誉资产归玩家所有，需可转移、可展示 |

---

### 11.2 存后端数据库即可的数据

这些数据**读写频繁、无需全局共识**，或仅用于平台内部展示，存链上会带来不必要的 Gas 消耗和延迟：

| 数据 | 存储方式 | 原因 |
|---|---|---|
| **`fill_ratio` 缓存** | 后端 DB（5 秒刷新） | 由链上 `fuel_quantity` 计算而来，缓存用于高频前端轮询，无需重复上链 |
| **比赛房间元信息** | 后端 DB | 包括房间名、创建时间、站点列表、当前参赛队数等展示字段 |
| **比赛状态机** | 后端 DB（`Lobby / PreStart / Running / Panic / Settling / Settled`）| 状态迁移高频（每秒检查），链上只需记录 start / end 两个关键节点 |
| **实时得分** | 后端内存 + DB | 比赛进行中每次 `FuelEvent` 触发后在后端累加，结算时快照上链即可 |
| **弹幕 / 通知消息** | 后端消息队列（WebSocket 推送）| 纯 UI 反馈，无资产属性，不需要链上留存 |
| **战队组成 / 角色分配** | 后端 DB | 组队是平台内部逻辑，只有"白名单地址"需要上链，角色信息存后端即可 |
| **玩家昵称 / 个人主页** | 后端 DB | 展示类信息，与资产无关 |
| **信誉分**（后续版本） | 后端 DB | v1 不上链；后续若需跨平台携带，再考虑迁移至链上 |
| **Optimistic UI 临时状态** | 前端内存 | 仅用于抵消链上延迟的视觉预更新，交易确认后自动同步链上数据 |

---

### 11.3 边界判断原则

遇到新需求时，用以下三个问题快速判断：

```
① 这个数据涉及真实资产的转移或归属吗？
   → 是 → 必须上链

② 这个数据是结算公正性的核心依据，需要任何人都能独立验证吗？
   → 是 → 必须上链

③ 这个数据主要用于平台内部展示 / 状态管理，且可以由链上数据推导出来吗？
   → 是 → 存后端，链上只保留源头数据
```

---

### 11.4 数据流向总览

```
EVE Frontier 链上（Sui）
│
├─ NetworkNode.fuel_quantity        ──读取──▶  后端缓存 fill_ratio
├─ FuelEvent(DEPOSITED)             ──监听──▶  后端归因计分 → WebSocket 推送
├─ deposit_fuel() tx sender         ──查询──▶  后端白名单比对
│
├─ start_match() / end_match()      ◀──写入──  后端比赛状态机（仅两个关键节点）
├─ 白名单写入                        ◀──写入──  队长 Lock & Pay 时
└─ LUX 奖励发放                     ◀──写入──  Settling → Settled 时

后端数据库（Off-chain）
│
├─ 比赛房间元信息、状态机
├─ 实时得分累加
├─ 战队 / 角色分配
├─ fill_ratio 缓存
└─ 玩家昵称、信誉分（v1）
```
