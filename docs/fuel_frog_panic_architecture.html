01系统总体架构

整体架构分为四层：前端层（Next.js Dashboard + In-game Overlay）、后端层（API Server + 三个后台服务）、数据持久层（Supabase / PostgreSQL） 以及 链上层（Sui / world::fuel）。选择 Supabase 作为核心数据层，因为它的 Realtime 功能原生支持 PostgreSQL Change → WebSocket 广播，完美契合弹幕、得分实时推送的需求，省去自建 WebSocket 服务的复杂性。

实时数据订阅（Supabase Realtime）

// lib/realtime.ts — 统一订阅管理
export function subscribeMatchUpdates(matchId: string) {
  return supabase
    .channel(`match-${matchId}`)
    // 得分变化 → 更新 matchStore.scores
    .on('postgres_changes', {
      event: 'UPDATE', schema: 'public', table: 'match_scores',
      filter: `match_id=eq.${matchId}`
    }, payload => matchStore.syncScores(payload.new))
    // 弹幕推送 → 插入 matchStore.danmaku
    .on('postgres_changes', {
      event: 'INSERT', schema: 'public', table: 'fuel_events',
      filter: `match_id=eq.${matchId}`
    }, payload => matchStore.pushDanmaku(payload.new))
    // 状态机迁移 → 切换页面/视觉模式
    .on('postgres_changes', {
      event: 'UPDATE', schema: 'public', table: 'matches',
      filter: `id=eq.${matchId}`
    }, payload => matchStore.handleStatusChange(payload.new.status))
    .subscribe()
}

03后端架构

后端分为两部分：Next.js API Routes（处理前端 REST 请求，轻量 BFF 层）和 独立 Node.js Workers（长期运行的后台进程，不适合放在 serverless 函数中）。Workers 通过 Supabase 客户端写入数据库，Supabase Realtime 自动向前端广播变化。

① Sui Event Listener Worker

1
订阅 Sui 事件流
suiClient.subscribeEvent({ filter: { MoveModule: { package, module: 'fuel' } } })
2
捕获 FuelEvent(DEPOSITED)
提取 assembly_id, old_quantity, new_quantity, tx_digest
3
调用 getTransactionBlock(tx_digest)
获取 sender（钱包地址）和 timestamp
4
三重过滤
Filter1: sender ∈ match.whitelist → Filter2: assembly_id ∈ match.target_nodes → Filter3: timestamp ∈ [start, end]
5
写入 Supabase
INSERT fuel_events + UPDATE match_scores → Realtime 自动推送前端

② Node Scanner Worker

// workers/nodeScanner.ts — 每 5 秒轮询全星系站点
async function scanNodes() {
  const nodes = await supabase.from('network_nodes').select('id, object_id')
  const results = await Promise.all(
    nodes.data.map(n => suiClient.getObject({ id: n.object_id, options: { showContent: true } }))
  )
  for (const obj of results) {
    const { fuel_quantity, fuel_max_capacity } = obj.data.content.fields
    const fill_ratio = fuel_quantity / fuel_max_capacity
    // 更新缓存 → Realtime 推送星系地图颜色变化
    await supabase.from('network_nodes')
      .update({ fill_ratio, updated_at: new Date() })
      .eq('object_id', obj.data.objectId)
    // 触发自动比赛生成
    if (fill_ratio < 0.20 && node.is_public) {
      await autoCreateMatch(node.id)
    }
  }
}
setInterval(scanNodes, 5000)

// 每秒检查所有 Running 比赛
async function tickStateMachine() {
  const running = await supabase.from('matches')
    .select('*').eq('status', 'running')

  for (const match of running.data) {
    const elapsed = Date.now() - match.started_at
    if (elapsed > MATCH_DURATION - PANIC_WINDOW) {  // 进入 Panic
      await supabase.from('matches').update({ status: 'panic' }).eq('id', match.id)
    }
    if (elapsed > MATCH_DURATION) {  // 结算
      await suiClient.executeTransaction(buildEndMatchTx(match.on_chain_id))
      await supabase.from('matches').update({ status: 'settling' }).eq('id', match.id)
    }
  }
}
setInterval(tickStateMachine, 1000)

06链上 / 链下数据边界与流向

数据归属一览
数据存储位置说明
deposit_fuel() + FuelEvent On-chain 唯一可信计分数据源，不可篡改
NetworkNode.fuel_quantity On-chain 原生链上对象
入场费 LUX 转账 On-chain 涉及真实资产，必须链上
白名单写入 On-chain 归因计分信任根
start_match / end_match On-chain 时间窗口锚点，可审计
奖励 LUX 发放 On-chain 结算合约 70/30 自动分账
fill_ratio 缓存 Off-chain DB 5s 轮询缓存，高频读
比赛状态机 Off-chain DB 每秒迁移，链上仅记录 start/end
实时得分 Off-chain DB 比赛中累加，结算时快照
弹幕/通知消息 Realtime WS 纯 UI 反馈，不持久化
Optimistic 临时状态 前端内存 交易确认后自动同步链上
战队/角色/昵称/信誉分 Off-chain DB 展示类信息，与资产无关
